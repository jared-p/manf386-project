92/* 



93 * Open a MAT-file "filename" using mode "mode".  Return



94 * a pointer to a MATFile for use with other MAT API functions.



95 *



96 * Current valid entries for "mode" are



97 * "r"    == read only.



98 * "w"    == write only (deletes any existing file with name <filename>).



99 * "w4"   == as "w", but create a MATLAB 4.0 MAT-file.



100 * "w7.3" == as "w", but create a MATLAB 7.3 MAT-file.



101 * "u"    == update.  Read and write allowed, existing file is not deleted.



102 * 



103 * Return NULL if an error occurs.



104 */



105EXTERN_C LIBMAT_API MATFile* matOpen(const char *filename, const char * mode);

186/*



187 * Read the array header for the variable with the specified name from 



188 * the MAT-file.



189 * 



190 * An array header contains all the same information as an



191 * array, except that the pr, pi, ir, and jc data structures are 



192 * not allocated for non-recursive data types.  That is,



193 * Cells, structures, and objects contain pointers to other



194 * array headers, but numeric, string, and sparse arrays do not 



195 * contain valid data in their pr, pi, ir, or jc fields.



196 *



197 * The purpose of an array header is to gain fast access to 



198 * information about an array without reading all the array's



199 * actual data.  Thus, functions such as mxGetM, mxGetN, and mxGetClassID



200 * can be used with array headers, but mxGetPr, mxGetPi, mxGetIr, mxGetJc,



201 * mxSetPr, mxSetPi, mxSetIr, and mxSetJc cannot.



202 *



203 * An array header should NEVER be returned to MATLAB (for example via the



204 * MEX API), or any other non-matrix access API function that expects a



205 * full mxArray (examples include engPutVariable(), matPutVariable(), and 



206 * mexPutVariable()).



207 *



208 * Return NULL if an error occurs.



209 */



210EXTERN_C LIBMAT_API mxArray * matGetVariableInfo(MATFile *pMF, const char * name);
