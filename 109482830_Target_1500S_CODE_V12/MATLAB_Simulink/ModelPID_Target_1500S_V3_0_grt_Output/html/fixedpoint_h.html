12/* Abstract:



13 *



14 * The primary motivation of this header file is to allow customers to



15 * write C language S-functions that support fixed-point data types.  However,



16 * the API in this header extends beyond fixed-point and includes support for



17 * standard floating point and integer data types.  



18 *



19 * This header contains definitions and prototypes for use in creating 



20 * Simulink S-functions that can process a wide variety of data types.



21 * These data types include the Simulink builtin data types,



22 *     double        single



23 *     uint8         int8



24 *     uint16        int16



25 *     uint32        int32



26 * all the MathWorks defined fixed-point data types such as



27 *     sfix16_En15



28 *     ufix32_En16



29 *     ufix128



30 *     sfix37_S3_B5



31 * and the versions of data types obtained when Data Type Override is set



32 * to Scaled Doubles such as



33 *     flts16



34 *     flts16_En15



35 *     fltu32_S3_B5



36 *



37 * This header file contains functions and definitions that allow S-functions



38 * to determine which of these data types an input port or output port is



39 * using.  The S-function can also specify which of these data types should



40 * be used for an input, output, DWork state, or Runtime Parameter.  The



41 * information required to properly read to and write from different variables



42 * of each data type is given.



43 */



44

84/* Simulink supports fixed-point data types with any number of bits from



85 *    1 to FXP_MAX_BITS for unsigned numbers and 



86 *    2 to FXP_MAX_BITS for signed numbers.



87 * The API in this header file treats pure integers just like



88 * any other fixed-point numbers.  Pure integers just happen 



89 * to have trivial scaling.  The 6 standard Simulink integer



90 * types uint8, int8, uint16, int16, uint32, and int32 are not 



91 * exceptions.  They are treated like fixed-point numbers 



92 * that happened to have trivial scaling and happened to have



93 * 8, 16, or 32 bits.  The same API applies.



94 *   Simulink builtin integers are special only in that they do



95 * not trigger the need for a Fixed-Point License. 



96 *



97 * The number of bits used to represent a signal in simulation



98 * may be more than the number specified.  In this case, the



99 * signals will be emulated inside various sized containers 



100 * according to the following rules.  For 32 or fewer bits, the



101 * rules are simple.  For 32 or fewer bits, the container is a



102 * scalar integer type directly understood by the compiler.  The



103 * case of using a scalar integer type is classified as single-word.



104 * The three cases with 32 or fewer bits are shown by a table.



105 *



106 *   Specified Bits       Container Bits   Container typedef



107 *    1 to  8              8               int8_T  or uint8_T



108 *    9 to 16             16               int16_T or uint16_T



109 *   17 to 32             32               int32_T or uint32_T



110 *



111 * For 33 or more bits, the rules are more complicated and can



112 * depend on the model, the Matlab Host, the current target



113 * for code generation, and the version of Simulink Fixed Point



114 * being used.  Signals with 33 or more bits will either be stored 



115 * using a single-word 



116 * container with more than 32 bits or will be stored using a 



117 * multi-word representation.



118 *



119 * All multi-word representations will follow a general form.



120 * 



121 * typedef struct {



122 *     SOME_UNSIGNED_CHUNK_TYPE chunks[ NUM_CHUNKS_FOR_M_BITS ];



123 * } (u)intM_T



124 *



125 * The chunk at index zero will hold the least significant bits, and



126 * the highest index will hold the most significant bits.



127 *



128 * Whether single-word or multi-word is used can depend on 



129 * the model, the Matlab Host, the current 



130 * target for code generation, and the version of Simulink Fixed Point.



131 * The specific type used for SOME_UNSIGNED_CHUNK_TYPE can also vary depending



132 * on the model, the Matlab Host, the current target for code generation,



133 * and the version of Simulink Fixed Point.



134 * 



135 * To be portable, sfunctions that are intended to support fixed-point



136 * signals with more than 32 bits must react to container variations



137 * at runtime.  The most general way to do this is to use the interfaces



138 * for getting and setting "bit-regions" provided by this header.



139 * To support use of code optimized to a specific additional APIs



140 * are provided that can query the current properties of a data type.



141 * However, sfunction authors need to be aware the properties of a data 



142 * type can vary depending on



143 * the model, the Matlab Host, the current 



144 * target for code generation, and the version of Simulink Fixed Point.



145 * A robustly written sfunction will need to check the properties at runtime



146 * at least once after update diagram has started.  The most portable sfunctions



147 * will be written so that they can use an optimized case if possible or 



148 * switch to general bit-region case if necessary.  At the very least,



149 * a robust sfunction will need to gracefully set an error if does not



150 * have a case that supports the current memory layout.  Keep in mind that



151 * supporting every specific case for the current release of Simulink Fixed Point



152 * does not guarantee that every specific case in a future release will be 



153 * covered.  For this reason, it is best to have bit-region case or at



154 * the very least gracefully error if needed.



155 *



156 * For signed numbers, the bit encoding format is always 



157 * two's complement.



158 *



159 * When the specified number of bits is less than the size of



160 * the container, the desired bits are always stored in the



161 * least significant part of the container.  Any "unused" bits



162 * are in the most significant part of the container.  



163 * 



164 * For both single word case and multiword cases, any unused bits 



165 * must be set to a sign extension.  If the data type is



166 * an unsigned number, then obviously, the number is always



167 * nonnegative so the (implicit) sign bit is zero.  For unsigned



168 * data types, all unused bits must always be cleared to zero.



169 *    If the data type is a signed number, then the sign bit is



170 * one for strictly negative numbers and zero otherwise.  If



171 * the stored integer is negative, then all unused bits must be



172 * set to one.  If the stored integer is positive, then all



173 * unused bits must be cleared to zero.



174 *



175 * Simulink also supports two builtin floating point data types.  



176 * These are the standard floating point singles and doubles.



177 *



178 * Simulink also supports a data type category that is a hybrid 



179 * between floating-point and fixed-point.  This category is



180 * called ScaledDouble.  ScaledDouble cases occur when



181 * an individual block is configured to have a fixed-point



182 * output, but the system it lives in has its Data Type Override 



183 * setting for a system is set to Scaled Doubles.  The



184 * resulting data type has the scaling the fixed-point



185 * specification, but stores its output using floating 



186 * point doubles.  Storing in a double means that overflow



187 * and precision issues are almost always eliminated. Removing the 



188 * range and precision issues is very useful for benchmarking, testing, 



189 * and debugging. 



190 *   ScaledDouble data types contain information



191 * signedness and number of bits of what their non-overrided data 



192 * types would have been.  This is useful for information for other



193 * portions of the model that are not being debugged with Data Type 



194 * Override.



195 */

221/* Simulink Scaling



222 *



223 * Simulink data types support scaling.  The Fixed-Point User's Guide



224 * presents the following general formula.



225 * 



226 *    V = S * Q + B



227 *



228 * where



229 *



230 *    V is the Real World Value in the engineering units of choice



231 *    Q is the called the Stored Integer Value.  It is the raw value 



232 *      stored in the digital hardware.  Despite the name Q can use



233 *      an integer or a floating point format.



234 *    S is the Slope.  For clarity, this is also called the Total-Slope



235 *    B is the Bias



236 *



237 * The Total-Slope is often decomposed into two parts.



238 *



239 *    S = F * 2^E



240 *



241 * where



242 *



243 *   1. <= F < 2.0



244 *



245 *    F is the Fractional Slope



246 *    E is the Fixed Exponent, E is always an integer.



247 *



248 * If



249 *    B == 0.0



250 * and



251 *    F == 1.0



252 * then



253 * there is a clean binary point interpretation.



254 *



255 *    Fraction Length = -1 * Fixed Exponent 



256 *



257 * For example, if 



258 *    S == 0.125



259 *    B == 0.0



260 * then



261 *    F == 1.0



262 *    E == -3



263 * so



264 *    Fraction Length = 3



265 * therefore, this data type has 3 bits to the right of the



266 * binary point.



267 *



268 * The concept of scaling is most useful for fixed-point cases, but



269 * it does generalize to all the scalar numeric data types.  For



270 * floating point doubles and singles and for pure integers, 



271 * the scaling is trivial



272 *    B == 0



273 *    S == 1



274 *    F == 1



275 *    E == 0



276 *    Fraction Length == 0



277 * The access methods to get scaling information can be used



278 * for all the data types, they just return trivial values



279 * when the scaling is trivial.



280 */ 



281



282#endif /* fxpsimulinkscalingintro_h */

341/* Overflow logging structure



342 *



343 * Some fixed-point math functions accept a pointer to this structure.



344 * These functions will initialize each of the event counts to zero.  Then



345 * the functions will carryout the requested math operations.  Each



346 * time an event is detect during the math operation the appropriate 



347 * count will be incremented.  The increment operations makes sure



348 * the count does not overflow to zero; it will saturate to the upper limit.



349 *   



350 * For example, suppose a fixed-point conversion function is called. Suppose,



351 * one overflow occurred during the conversion.  After the function return,



352 * the memory pointed to be overflow logging structure pointer would indicate



353 * one overflow, zero saturations, and zero divide-by-zeros.



354 */



355typedef struct fxpOverflowLogs_tag



356{



357      int OverflowOccurred;



358      int SaturationOccurred;



359      int DivisionByZeroOccurred;



360



361} fxpOverflowLogs;

395 * the data types of input ports, output ports, RunTimeParameters, and DWork



396 * states.  The Data Type Id can also be used with all the standard 



397 * data type access methods in simstruc.h such as ssGetDataTypeSize. 



398 *    



399 * The input arguments are



400 *    isSigned              TRUE if signed, FALSE if unsigned



401 *    wordLength            total number of bits including any sign bit



402 *    fractionLength        number of bit to right of binary point



403 *    obeyDataTypeOverride  if FALSE ignore system's setting for Data Type Override



404 *                          if TRUE obey Data Type Override setting, depending



405 *                          on Data Type Override, resulting data type could be



406 *                          True Double, True Single, Scaled Double, or the 



407 *                          requested fixed point type.



408 *



409 * Cautions:



410 *



411 * 1) If the registered data type is not one of the builtin data types, then



412 * a Fixed-Point License will be checked out.  To prevent, a Fixed-Point



413 * License from being checked out when a user simply opens or views a model,



414 * calls to registration should be protected with



415 *    if ( ssGetSimMode(S) != SS_SIMMODE_SIZES_CALL_ONLY )



416 *       ssRegisterDataType ...



417 * 



418 * 2) There is no fixed relationship between the Data Type Id value and



419 * the input arguments.  Simulink hands out data type Ids on a first come, first



420 * served basis, so small changes to a model can cause the a different data



421 * type id value to be returned.  Always uses functions to get data type



422 * attributes from the data type id; never directly rely on the data type



423 * id value.
